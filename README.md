# Spice86 Ghidra Plugin

## Intention
The plugin is meant to make Ghidra usable enough for old DOS programs that Spice86 can then
consume generated overrides from that corrected model.

The intended workflow is:
- import Spice86 runtime evidence into Ghidra
- use that evidence to improve Ghidra's view of the program
- generate Spice86 C# overrides from the resulting listing

Spice86 contributes information that static DOS analysis often misses:
- which addresses really executed
- dynamic call and jump targets
- observed return targets
- runtime-only code regions
- recorded function entrypoints from the dump set

The restored plugin keeps that workflow, but fixes the address-model mismatch between Spice86
runtime physical addresses and Ghidra's normal DOS EXE load addresses.

## Menu Actions
The plugin adds:

- `Spice86 -> Import runtime evidence`
- `Spice86 -> Generate C#`

`Import runtime evidence` reads the dump set pointed to by `SPICE86_DUMPS_FOLDER` and imports:
- recorded function names from `spice86dumpGhidraSymbols.txt`
- dynamic call/jump references from `spice86dumpExecutionFlow.json`
- observed return targets
- executed block bookmarks from `ExecutedInstructions`
- executable-byte-write bookmarks when `ExecutableAddressWrittenBy` is present

`Generate C#` runs the override generator on the current program after applying the same runtime to
Ghidra address translation used by the importer.

## Address Model
Spice86 records runtime physical addresses such as `segment * 0x10 + offset`, while Ghidra's DOS
loader normally stores the program at a different linear base. The plugin now infers the linear
delta between those two models and uses it consistently for both importing and generating. That is
the main compatibility fix needed to make the original workflow work again on a normal DOS EXE
import.

## What this restore deliberately does not add
This restore is meant to make the original workflow work again, not to invent new repair features.
In particular:
- it does not patch bytes from the final memory dump into the program
- it does not add a new runtime-memory repair phase
- it does not repeatedly decompile, split, or recreate functions aggressively

The importer remains conservative:
- it does not delete existing references wholesale
- it only disassembles at observed entrypoints and targets when that is safe
- it only renames functions when Ghidra is still using an autogenerated name

## Requirements
- Java 21-compatible build target
- Ghidra 12.0.4_PUBLIC tested

Other Ghidra versions may work, but the extension zip should be built against the exact version you want to install.

## Building
Set `GHIDRA_INSTALL_DIR` to your local Ghidra directory and run:

```bash
GRADLE_USER_HOME=/tmp/ghidra-gradle \
JAVA_TOOL_OPTIONS=-Duser.home=/tmp/ghidra-home \
GHIDRA_INSTALL_DIR=/path/to/ghidra_12.0.4_PUBLIC \
/path/to/ghidra_12.0.4_PUBLIC/support/gradle/gradlew buildExtension
```

The zip will be written to `dist/`.

## Installing
Install the generated zip via:

`File -> Install Extensions`

Then restart Ghidra.

## Dump setup
Before running the plugin, set:

```bash
export SPICE86_DUMPS_FOLDER=/path/to/dump/folder
```

The folder is expected to contain at least:
- `spice86dumpExecutionFlow.json`
- `spice86dumpGhidraSymbols.txt` (optional but recommended)
- `CodeGeneratorConfig.json` (optional, for generator customization)

## Headless usage
You can run the import + generation flow headlessly with:

```bash
SPICE86_DUMPS_FOLDER=/path/to/dump/folder \
/path/to/ghidra_12.0.4_PUBLIC/support/analyzeHeadless \
  /tmp/ghidra-project Spice86Import \
  -import /path/to/program.exe \
  -noanalysis \
  -scriptPath /path/to/spice86-ghidra-plugin/src/main/java \
  -postScript Spice86PluginRunner.java \
  -deleteProject
```

## Verified locally
Tested against Ghidra `12.0.4_PUBLIC` by:
- compiling all Java sources against the extracted Ghidra jars
- building the extension zip with Ghidra's bundled Gradle wrapper
- running `Spice86PluginRunner.java` headlessly on `Spice_analysis/DS`
- running `Spice86PluginRunner.java` headlessly on `Spice_analysis/DUNE`

Those headless runs imported runtime evidence and generated C# override files. Summary logs were
written to:

`Spice_analysis/DS/Spice86RuntimeEvidence.txt`
`Spice_analysis/DS/Spice86CodeGenerator.txt`
`Spice_analysis/DUNE/Spice86RuntimeEvidence.txt`
`Spice_analysis/DUNE/Spice86CodeGenerator.txt`
